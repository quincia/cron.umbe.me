Architettura Event-Driven in Pimcore: Implementare CQRS e Event Sourcing per Applicazioni Enterprise Scalabili

Pimcore è una piattaforma potente, ma la sua architettura monolitica può diventare un collo di bottiglia quando gestisci milioni di DataObject, decine di tenant e operazioni batch che bloccano l’intera applicazione. Ho visto installazioni enterprise rallentare drasticamente durante importazioni massive perché ogni salvataggio di un Asset attiva listener sincroni, rigenera cache e aggiorna indici — tutto nello stesso thread HTTP.

Questo articolo mostra come superare questi limiti implementando un’architettura Event-Driven con pattern CQRS e Event Sourcing. Non parliamo di teoria: vedrai codice production-ready, configurazioni testate e le trappole in cui sono caduto io stesso.

Il risultato finale: un sistema che gestisce 50.000+ operazioni/ora mantenendo latenza sotto i 100ms per le letture, con tracciabilità completa di ogni modifica su ogni DataObject.

Prerequisiti

Stack tecnico richiesto:

  • Pimcore 11.x con Symfony 6.4
  • PHP 8.2+ con estensioni pcntl, redis
  • RabbitMQ 3.12+ o Redis 7+ (per Streams)
  • Elasticsearch 8.x
  • MySQL 8.0+ con replica read

Conoscenze necessarie:

  • Familiarità con Symfony Messenger e i suoi concetti (envelope, stamp, transport)
  • Comprensione del modello DataObject di Pimcore e dei suoi eventi
  • Esperienza con Doctrine e gestione delle transazioni

Concetti fondamentali:

CQRS (Command Query Responsibility Segregation): separiamo fisicamente le operazioni di scrittura (Commands) dalle letture (Queries). Le scritture passano attraverso Pimcore e MySQL, le letture vanno direttamente su Elasticsearch o read replicas.

Event Sourcing: invece di salvare solo lo stato corrente, registriamo ogni evento che ha modificato un DataObject. Questo ci dà versioning illimitato, audit trail completo e la possibilità di ricostruire lo stato a qualsiasi punto nel tempo.

Eventual Consistency: accettiamo che i dati su Elasticsearch possano essere “indietro” di qualche secondo rispetto a MySQL. In cambio otteniamo scalabilità orizzontale delle letture.

# config/packages/messenger.yaml
framework:
    messenger:
        # Serializzazione custom per gestire oggetti Pimcore
        serializer:
            default_serializer: messenger.transport.symfony_serializer
            symfony_serializer:
                format: json
                context: { }

        # Definiamo i bus separati per CQRS
        buses:
            command.bus:
                middleware:
                    - doctrine_transaction  # Wrap in transazione
                    - App\Messenger\Middleware\TenantAwareMiddleware
                    - App\Messenger\Middleware\AuditLogMiddleware
                default_middleware: allow_no_handlers

            event.bus:
                default_middleware: allow_no_handlers
                middleware:
                    - App\Messenger\Middleware\TenantAwareMiddleware

            query.bus:
                default_middleware: allow_no_handlers

        transports:
            # Comandi ad alta priorità - esecuzione quasi immediata
            commands_high:
                dsn: '%env(RABBITMQ_DSN)%'
                options:
                    exchange:
                        name: pimcore_commands
                        type: direct
                    queues:
                        commands_high:
                            binding_keys: [high]
                retry_strategy:
                    max_retries: 3
                    delay: 1000
                    multiplier: 2

            # Comandi standard - batch operations
            commands_low:
                dsn: '%env(RABBITMQ_DSN)%'
                options:
                    exchange:
                        name: pimcore_commands
                        type: direct
                    queues:
                        commands_low:
                            binding_keys: [low]
                retry_strategy:
                    max_retries: 5
                    delay: 5000
                    multiplier: 3

            # Eventi di dominio - fan-out a tutti i consumer
            domain_events:
                dsn: '%env(RABBITMQ_DSN)%'
                options:
                    exchange:
                        name: pimcore_domain_events
                        type: fanout
                    queues:
                        projector_elasticsearch: ~
                        projector_cache: ~
                        notification_handler: ~

            # Operazioni async pesanti (import, export, report)
            async_heavy:
                dsn: '%env(REDIS_DSN)%/messages_heavy'
                options:
                    stream: pimcore_heavy_jobs
                    group: workers
                    consumer: '%env(HOSTNAME)%'

            # Dead letter queue per messaggi falliti
            failed:
                dsn: 'doctrine://default?queue_name=failed_messages'

        failure_transport: failed

        routing:
            # Routing dei comandi
            'App\Command\DataObject\*': commands_high
            'App\Command\Batch\*': commands_low
            'App\Command\Import\*': async_heavy

            # Routing degli eventi
            'App\Event\Domain\*': domain_events

⚠️ Attenzione: Non usare mai sync:// come fallback in produzione. Se RabbitMQ non è raggiungibile, l’applicazione deve fallire esplicitamente, non processare messaggi in modo sincrono rallentando tutto.

<?php
// src/Messenger/Middleware/TenantAwareMiddleware.php

declare(strict_types=1);

namespace App\Messenger\Middleware;

use App\Messenger\Stamp\TenantStamp;
use App\Service\TenantResolver;
use Pimcore\Model\DataObject;
use Symfony\Component\Messenger\Envelope;
use Symfony\Component\Messenger\Middleware\MiddlewareInterface;
use Symfony\Component\Messenger\Middleware\StackInterface;
use Symfony\Component\Messenger\Stamp\ReceivedStamp;

/**
 * Middleware che propaga il contesto tenant attraverso i messaggi asincroni.
 * Fondamentale per scenari multi-tenant dove ogni operazione deve essere
 * isolata nel proprio contesto dati.
 */
final class TenantAwareMiddleware implements MiddlewareInterface
{
    public function __construct(
        private readonly TenantResolver $tenantResolver,
    ) {
    }

    public function handle(Envelope $envelope, StackInterface $stack): Envelope
    {
        // Se il messaggio arriva da un transport (consumer), ripristiniamo il tenant
        if ($envelope->last(ReceivedStamp::class) !== null) {
            $tenantStamp = $envelope->last(TenantStamp::class);
            
            if ($tenantStamp instanceof TenantStamp) {
                // Impostiamo il tenant context per Pimcore
                $this->tenantResolver->setCurrentTenant($tenantStamp->getTenantId());
                
                // Configuriamo anche l'inheritance per i DataObject
                DataObject::setHideUnpublished($tenantStamp->hideUnpublished());
                DataObject::setGetInheritedValues($tenantStamp->inheritValues());
            }
            
            return $stack->next()->handle($envelope, $stack);
        }

        // Se stiamo inviando un messaggio, aggiungiamo il tenant corrente
        $currentTenant = $this->tenantResolver->getCurrentTenant();
        
        if ($currentTenant !== null && $envelope->last(TenantStamp::class) === null) {
            $envelope = $envelope->with(new TenantStamp(
                tenantId: $currentTenant->getId(),
                hideUnpublished: DataObject::getHideUnpublished(),
                inheritValues: DataObject::getGetInheritedValues(),
            ));
        }

        return $stack->next()->handle($envelope, $stack);
    }
}

<?php
// src/EventSourcing/Event/DataObjectEvent.php

declare(strict_types=1);

namespace App\EventSourcing\Event;

use DateTimeImmutable;
use Ramsey\Uuid\UuidInterface;

/**
 * Classe base per tutti gli eventi di dominio relativi ai DataObject.
 * Immutabile per design - un evento rappresenta qualcosa che È SUCCESSO.
 */
abstract readonly class DataObjectEvent
{
    public function __construct(
        private UuidInterface $eventId,
        private int $aggregateId,      // ID del DataObject Pimcore
        private string $aggregateType, // Classe del DataObject
        private int $version,          // Versione progressiva dell'aggregato
        private string $tenantId,
        private ?int $userId,
        private DateTimeImmutable $occurredAt,
        private array $metadata = [],
    ) {
    }

    public function getEventId(): UuidInterface
    {
        return $this->eventId;
    }

    public function getAggregateId(): int
    {
        return $this->aggregateId;
    }

    public function getAggregateType(): string
    {
        return $this->aggregateType;
    }

    public function getVersion(): int
    {
        return $this->version;
    }

    public function getTenantId(): string
    {
        return $this->tenantId;
    }

    public function getUserId(): ?int
    {
        return $this->userId;
    }

    public function getOccurredAt(): DateTimeImmutable
    {
        return $this->occurredAt;
    }

    public function getMetadata(): array
    {
        return $this->metadata;
    }

    /**
     * Ogni evento deve definire il proprio tipo univoco.
     * Usato per la deserializzazione e il routing.
     */
    abstract public static function getEventType(): string;

    /**
     * Payload specifico dell'evento.
     */
    abstract public function getPayload(): array;
}

<?php
// src/EventSourcing/Event/DataObjectPropertyChanged.php

declare(strict_types=1);

namespace App\EventSourcing\Event;

use DateTimeImmutable;
use Ramsey\Uuid\UuidInterface;

/**
 * Evento emesso quando una o più proprietà di un DataObject cambiano.
 * Registra solo i delta, non l'intero oggetto.
 */
final readonly class DataObjectPropertyChanged extends DataObjectEvent
{
    /**
     * @param array<string, array{old: mixed, new: mixed}> $changes
     */
    public function __construct(
        UuidInterface $eventId,
        int $aggregateId,
        string $aggregateType,
        int $version,
        string $tenantId,
        ?int $userId,
        DateTimeImmutable $occurredAt,
        private array $changes,
        private string $changeSource, // 'api', 'admin', 'import', 'workflow'
        array $metadata = [],
    ) {
        parent::__construct(
            $eventId,
            $aggregateId,
            $aggregateType,
            $version,
            $tenantId,
            $userId,
            $occurredAt,
            $metadata
        );
    }

    public static function getEventType(): string
    {
        return 'data_object.property_changed';
    }

    public function getPayload(): array
    {
        return [
            'changes' => $this->changes,
            'change_source' => $this->changeSource,
        ];
    }

    public function getChanges(): array
    {
        return $this->changes;
    }

    public function getChangeSource(): string
    {
        return $this->changeSource;
    }

    /**
     * Verifica se una specifica proprietà è cambiata.
     */
    public function hasPropertyChanged(string $propertyName): bool
    {
        return isset($this->changes[$propertyName]);
    }

    /**
     * Ottiene il valore precedente di una proprietà.
     */
    public function getOldValue(string $propertyName): mixed
    {
        return $this->changes[$propertyName]['old'] ?? null;
    }

    /**
     * Ottiene il nuovo valore di una proprietà.
     */
    public function getNewValue(string $propertyName): mixed
    {
        return $this->changes[$propertyName]['new'] ?? null;
    }
}

<?php
// src/EventSourcing/Store/DoctrineEventStore.php

declare(strict_types=1);

namespace App\EventSourcing\Store;

use App\EventSourcing\Event\DataObjectEvent;
use App\EventSourcing\Event\DataObjectPropertyChanged;
use App\EventSourcing\Exception\ConcurrencyException;
use App\EventSourcing\Serializer\EventSerializer;
use Doctrine\DBAL\Connection;
use Doctrine\DBAL\Exception\UniqueConstraintViolationException;

/**
 * Event Store basato su MySQL con supporto per optimistic concurrency control.
 * La tabella usa una chiave composta (aggregate_id, version) per garantire
 * l'ordine degli eventi e prevenire write conflicts.
 */
final class DoctrineEventStore implements EventStoreInterface
{
    private const TABLE_NAME = 'event_store';

    public function __construct(
        private readonly Connection $connection,
        private readonly EventSerializer $serializer,
    ) {
    }

    /**
     * Appende un evento allo store.
     * 
     * @throws ConcurrencyException Se la versione attesa non corrisponde
     */
    public function append(DataObjectEvent $event, int $expectedVersion): void
    {
        // Verifichiamo che la versione sia quella attesa (optimistic locking)
        if ($event->getVersion() !== $expectedVersion + 1) {
            throw new ConcurrencyException(sprintf(
                'Versione attesa %d, ricevuta %d per aggregato %d',
                $expectedVersion + 1,
                $event->getVersion(),
                $event->getAggregateId()
            ));
        }

        try {
            $this->connection->insert(self::TABLE_NAME, [
                'event_id' => $event->getEventId()->toString(),
                'aggregate_id' => $event->getAggregateId(),
                'aggregate_type' => $event->getAggregateType(),
                'version' => $event->getVersion(),
                'event_type' => $event::getEventType(),
                'payload' => $this->serializer->serialize($event),
                'metadata' => json_encode($event->getMetadata(), JSON_THROW_ON_ERROR),
                'tenant_id' => $event->getTenantId(),
                'user_id' => $event->getUserId(),
                'occurred_at' => $event->getOccurredAt()->format('Y-m-d H:i:s.u'),
                'created_at' => (new \DateTimeImmutable())->format('Y-m-d H:i:s.u'),
            ]);
        } catch (UniqueConstraintViolationException $e) {
            // Qualcun altro ha scritto prima di noi - concurrency conflict
            throw new ConcurrencyException(
                'Conflitto di concorrenza: evento già presente per questa versione',
                previous: $e
            );
        }
    }

    /**
     * Recupera tutti gli eventi per un aggregato, opzionalmente da una versione specifica.
     * 
     * @return iterable<DataObjectEvent>
     */
    public function getEventsForAggregate(
        int $aggregateId,
        int $fromVersion = 0,
        ?int $toVersion = null
    ): iterable {
        $qb = $this->connection->createQueryBuilder()
            ->select('*')
            ->from(self::TABLE_NAME)
            ->where('aggregate_id = :aggregateId')
            ->andWhere('version > :fromVersion')
            ->setParameter('aggregateId', $aggregateId)
            ->setParameter('fromVersion', $fromVersion)
            ->orderBy('version', 'ASC');

        if ($toVersion !== null) {
            $qb->andWhere('version <= :toVersion')
               ->setParameter('toVersion', $toVersion);
        }

        $result = $qb->executeQuery();

        while ($row = $result->fetchAssociative()) {
            yield $this->serializer->deserialize($row['payload'], $row['event_type']);
        }
    }

    /**
     * Ottiene la versione corrente di un aggregato.
     */
    public function getCurrentVersion(int $aggregateId): int
    {
        $result = $this->connection->createQueryBuilder()
            ->select('MAX(version) as max_version')
            ->from(self::TABLE_NAME)
            ->where('aggregate_id = :aggregateId')
            ->setParameter('aggregateId', $aggregateId)
            ->executeQuery()
            ->fetchOne();

        return (int) ($result ?? 0);
    }

    /**
     * Ricostruisce lo stato di un DataObject a una versione specifica.
     * Utile per debug, audit e "time travel".
     */
    public function getSnapshotAtVersion(int $aggregateId, int $version): array
    {
        $state = [];
        
        foreach ($this->getEventsForAggregate($aggregateId, 0, $version) as $event) {
            if ($event instanceof DataObjectPropertyChanged) {
                foreach ($event->getChanges() as $property => $change) {
                    $state[$property] = $change['new'];
                }
            }
            // Gestisci altri tipi di evento...
        }

        return $state;
    }
}

💡 Tip: Per aggregati con migliaia di eventi, implementa uno snapshot store separato. Ogni N eventi (es. 100), salva uno snapshot completo e riparti da lì invece che dall’inizio.
