Introduzione

Un’immagine Docker da 1.2GB per un’applicazione Go da 10MB: suona familiare? Questo è uno dei problemi più comuni e costosi nel mondo dei container. Tempi di pull lunghi, storage sprecato, superficie d’attacco inutilmente grande.

I multi-stage builds risolvono questo problema in modo elegante, permettendoti di usare un’immagine ricca di tool per la build e un’immagine minimale per la produzione, tutto in un singolo Dockerfile.

Prerequisiti

  • Docker installato (versione 17.05+)
  • Conoscenza base dei Dockerfile
  • Un’applicazione da containerizzare (usiamo Go e Node.js come esempi)

# ✅ Multi-stage: separare build e runtime

# Stage 1: Builder
FROM golang:1.21-alpine AS builder

WORKDIR /app

# Copia solo i file per le dipendenze prima (cache layer ottimizzato)
COPY go.mod go.sum ./
RUN go mod download

# Copia il codice sorgente
COPY . .

# Build con ottimizzazioni per produzione
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-w -s" \
    -o /server .

# Stage 2: Runtime (immagine minimale)
FROM scratch

# Copia solo il binario compilato
COPY --from=builder /server /server

# Copia i certificati SSL se necessario
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/server"]

Risultato: ~8MB (solo il binario).

# Stage 1: Install e compile
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build  # TypeScript compile

# Stage 3: Runtime
FROM node:20-alpine AS runner
WORKDIR /app

ENV NODE_ENV=production

# Solo le dipendenze di produzione
COPY --from=deps /app/node_modules ./node_modules
# Solo il codice compilato
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json ./package.json

EXPOSE 3000
USER node
CMD ["node", "dist/index.js"]

Errori comuni e come evitarli

⚠️ Ordine dei layer: Copia sempre package.json e installa le dipendenze PRIMA di copiare il codice. Questo sfrutta la cache Docker.

# ❌ Invalida la cache ogni volta che il codice cambia
COPY . .
RUN npm install

# ✅ Cache stabile per le dipendenze
COPY package*.json ./
RUN npm install
COPY . .

💡 Non copiare il .dockerignore: Crea sempre un file .dockerignore per escludere node_modules, .git, *.log e ambienti locali.

Best Practices

1. Usa tag specifici, mai latest

1
2

FROM node:20.10.0-alpine  # ✅ Riproducibile
FROM node:latest           # ❌ Non deterministico

2. Un processo per container

1
2

CMD ["node", "server.js"]     # ✅ Un processo
CMD ["./start.sh"]            # ⚠️ Script wrapper, ok ma attenzione ai segnali

3. Non girare come root

1
2

RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

4. Sfrutta BuildKit per performance

1
2
3

DOCKER_BUILDKIT=1 docker build .
# oppure aggiungi in /etc/docker/daemon.json:
# { "features": { "buildkit": true } }

Conclusioni e Next Steps

I multi-stage builds sono uno strumento fondamentale per chiunque lavori con Docker in produzione. I benefici principali sono:

  • Riduzione dimensioni dell'80-95%
  • Sicurezza migliorata (meno tool = meno superficie d’attacco)
  • Build più veloci nei CI/CD pipeline
  • Stesso Dockerfile per sviluppo e produzione

Come prossimi passi, esplora:

  • Docker BuildKit e cache mount
  • Distroless images di Google
  • Dive: tool per analizzare i layer delle immagini
